# Why Is My INT8 Model Slower? A Visual Investigation

## The Problem

You've heard that INT8 quantization makes AI models faster. You quantize your model. You benchmark it. **It's slower.** What went wrong?

This paper runs the same matrix multiplication (the core operation in every AI model) across three different tools — **PyTorch**, **CUDA C**, and **ONNX Runtime** — and measures exactly what happens. We record the results to CSV files, plot them, and explain every result.

**The surprising findings:**

- CUDA C INT8 is **3-4x faster** than FP32
- PyTorch INT8 is **2x slower** than FP32
- ONNX INT8 on **CPU beats GPU** for small matrices

Same math. Same GPU. Completely different results. This paper explains why.

---

## The Experiment

### What We're Measuring

A single matrix multiplication: `output = input × weights + bias`

This is what every layer in an AI model does — GPT, LLaMA, Stable Diffusion, all of them. We vary the matrix size to simulate different model layers:

| Name | Size | Weight Memory | Simulates |
|------|------|--------------|-----------|
| Small | 4096 × 4096 | 64 MB | GPT-2 layer |
| Medium | 4096 × 11008 | 172 MB | LLaMA-7B FFN |
| Large | 8192 × 8192 | 256 MB | LLaMA-13B layer |

We test with **batch=1** (one input at a time, like chatbot generation) and **batch=32** (processing many inputs together, like batch inference).

### The Hardware

```
GPU: NVIDIA RTX A3000 Laptop GPU
  - 4,096 CUDA Cores (general math)
  - 128 Tensor Cores (specialized matrix math)
  - 5.7 GB memory
  - ~192 GB/s memory bandwidth
```

### The Three Tools

1. **PyTorch** — the most popular AI framework. Uses `torchao` for INT8 quantization.
2. **CUDA C** — raw GPU programming. Calls NVIDIA's math library (cuBLAS) directly.
3. **ONNX Runtime** — Microsoft's inference engine. Runs on CPU and GPU.

## Why Does This Happen?

### Question 1: Why is CUDA C INT8 the fastest?

![Data Paths](plots/06_data_paths.png)

**CUDA C does real INT8 math. PyTorch doesn't.**

When CUDA C multiplies INT8 numbers, it uses a special GPU instruction called **IMMA** (Integer Matrix Multiply Accumulate). This instruction takes two INT8 numbers and multiplies them directly — no conversion needed.

PyTorch's `torchao` does something different. It stores the weights as INT8 to save memory, but **converts them back to FP32 before doing the actual math**. It's like zipping a file to save disk space but unzipping it every time you open it.

The conversion step (dequantization) takes **55% of the total GPU time**. So PyTorch spends more than half its time just converting numbers, not doing useful math.

### Question 2: Why is PyTorch INT8 slower than FP32?

The profiler tells us exactly where the GPU spends its time:

![Core Type Breakdown](plots/03_core_breakdown.png)

*Data source: `results/profiling.csv`*

Look at the "INT8 torchao" bars:
- **batch=1**: 55.6% of time on Memory/Elem (the dequantization step), only 41.6% on actual compute
- **batch=32**: 55.2% on Memory/Elem, only 42.1% on compute

More than half the GPU time is wasted converting INT8 back to FP32. Plain FP32 doesn't need this step, so it's faster.

**Then why does torchao exist?** It's a **memory** optimization, not a speed optimization. A 70B parameter model needs 280 GB at FP32 — it won't fit on any GPU. At INT8, it's 70 GB — fits on one A100. You trade speed for the ability to run the model at all.

### Question 3: Why does ONNX INT8 on CPU beat GPU for small matrices?

At 4K×4K, ONNX INT8 CPU takes **0.097 ms** while PyTorch FP32 GPU takes **0.281 ms**. The CPU is 2.9x faster. Three reasons:

1. **Real INT8 math on CPU**: ONNX Runtime uses Intel's VNNI instructions that do INT8×INT8 natively. Like CUDA C's Tensor Cores, but on the CPU.

2. **No GPU startup cost**: Every GPU operation has a ~5-15 microsecond launch overhead. For a 0.08 ms computation, that's significant. CPU just runs — no launch, no transfer.

3. **The weight fits in CPU cache**: The 16 MB INT8 weight matrix (4K×4K) fits in the CPU's L3 cache (~12-30 MB). Data from cache is much faster than data from GPU memory. At 8K×8K (64 MB), it doesn't fit in cache anymore, and the CPU advantage vanishes.

### Question 4: Why does TF32 ON/OFF make no difference at batch=1?

Look at the profiling data at batch=1: both FP32 TF32=OFF and FP32 TF32=ON show **identical** results — 95.2% CUDA Core. Both run the same `gemvx` kernel.

At batch=32, they're different: TF32=OFF uses CUDA Cores (95.1%), TF32=ON uses Tensor Cores (95.2%).

**Why?** At batch=1, the operation is a matrix-**vector** multiply (GEMV). NVIDIA's math library has no Tensor Core implementation for GEMV — it always uses CUDA Cores. The TF32 setting is simply ignored.

At batch=32, it's a matrix-**matrix** multiply (GEMM), and Tensor Cores can be used.

This matters for chatbots: when generating one token at a time (batch=1), your expensive Tensor Cores sit idle.

## What Should You Use?

| Situation | Best choice | Why |
|-----------|------------|-----|
| Maximum speed on GPU | CUDA C INT8 (cuBLASLt) | True INT8 math on Tensor Cores |
| Small model, no GPU | ONNX INT8 on CPU | Native VNNI, fits in cache |
| Big model that won't fit in GPU | PyTorch torchao INT8 | 4x compression to fit in memory |
| Cross-platform deployment | ONNX Runtime | Same model runs on CPU/GPU/mobile |
| Just prototyping | PyTorch FP32 | Simplest, no quantization complexity |

## Optimization: Closing the Gap with CUDA C

Since PyTorch torchao INT8 was 2.3x *slower* than FP32, we tested every available optimization to find the fastest PyTorch path. Results on 4Kx4K (b=1):

| Method | Time | vs FP32 | What it does |
|--------|------|---------|-------------|
| **FP16 (Tensor Core HMMA)** | **0.140 ms** | **1.9x faster** | `model.half()` — uses Tensor Core FP16 matmul |
| FP32 baseline | 0.269 ms | 1.0x | Standard `nn.Linear` with TF32 |
| CUDA C INT8 IGEMM | 0.098 ms | 2.7x faster | cuBLASLt native INT8×INT8 (unreachable from PyTorch) |
| INT8 Weight-Only (torchao) | 0.633 ms | 2.4x slower | Dequant INT8→FP32, then FP32 GEMM |
| INT8 Dynamic Act+Weight | 2.445 ms | 9.1x slower | Quantize activations at runtime + INT8 GEMM |
| torch.compile | 1.852 ms | 6.9x slower | Inductor overhead dominates at small batch |

For batch=32, `torch._int_mm` (native INT8 Tensor Core) becomes available:

| Method | Time | vs FP32 |
|--------|------|---------|
| **FP16** | **0.378 ms** | **2.0x faster** |
| torch._int_mm (raw INT8 TC) | 0.535 ms | 1.4x faster |
| FP32 baseline | 0.739 ms | 1.0x |
| CUDA C INT8 IGEMM | 0.220 ms | 3.4x faster |

**Key findings:**

1. **FP16 is the practical winner in PyTorch.** One line (`model.half()`) gives 2x speedup by routing matmul to Tensor Core HMMA. No quantization complexity needed.

2. **`torch._int_mm` only works for batch >= 16.** This is PyTorch's native INT8×INT8→INT32 Tensor Core op, but it requires M (batch dimension) >= 16. For batch=1 inference (chatbot generation), it's unusable.

3. **CUDA C INT8 remains 2-3x faster than any PyTorch INT8 path.** cuBLASLt handles batch=1 natively (no M >= 16 restriction). The gap is in the library, not the hardware.

4. **torch.compile adds overhead for small matrices.** The Inductor's kernel launch and graph overhead exceeds the compute savings at these sizes.

### Practical Recommendation

```
┌─────────────────────────────────────────────────────────────────┐
│  IF you want fast inference in PyTorch:                         │
│    → model.half()  (FP16 on Tensor Cores, 2x faster)           │
│                                                                 │
│  IF you want maximum speed:                                     │
│    → CUDA C cuBLASLt INT8 IGEMM  (3-4x faster, requires C)    │
│                                                                 │
│  IF you want real-world model serving:                          │
│    → Ollama / llama.cpp  (optimized fused dequant+GEMM)        │
│                                                                 │
│  DO NOT use torchao INT8 weight-only for speed.                 │
│    It's 2x slower than FP32. Use it only for memory savings.   │
└─────────────────────────────────────────────────────────────────┘
```

## Summary

1. **"INT8 is faster" is only true if you do real INT8 math.** CUDA C does. PyTorch torchao doesn't — it converts back to FP32.
2. **FP16 is the fastest practical option in PyTorch.** One line of code, 2x speedup, no quantization complexity.
3. **The GPU has two types of math units.** CUDA Cores (general) and Tensor Cores (specialized). Tensor Cores only activate for batch >= 2.
4. **CPU can beat GPU for small matrices** because of cache effects and zero launch overhead.
5. **4x model compression is the guaranteed win.** Every INT8 approach gives exactly 4x smaller models.
6. **~1% accuracy loss is typical.** Acceptable for inference, not for training.
7. **torch._int_mm (native INT8 Tensor Core) requires batch >= 16.** This makes it unusable for single-token generation (the most common LLM inference pattern).

## Real-World Context: Running Quantized Models via Ollama

The synthetic benchmarks above explain *why* performance differs at the kernel level. But what happens when you run a *real* quantized model through a serving framework?

We tested two models via **Ollama** on the same RTX A3000 GPU:

| Model | Quantization | Size | Inference | GPU Offload | VRAM Used |
|-------|-------------|------|-----------|-------------|-----------|
| llama3.2:3b | **Q4_K_M (4-bit)** | 1.9 GB | ~132 ms | 100% GPU | 3.1 GB |
| MedGemma1.5:4b | F16 (full) | 7.4 GB | ~336 ms | 60% GPU | 5.6 GB |

**Key observations:**

1. **Q4_K_M is a different quantization scheme than INT8.** It uses 4-bit mixed precision (some blocks at higher precision for important weights). This is what GGUF models use in practice.

2. **The quantized model (llama3.2 Q4_K_M) is 2.5x faster** than the full-precision model (MedGemma F16). This matches the synthetic benchmark pattern: less data to move = faster inference.

3. **VRAM is the bottleneck, not compute.** The Q4_K_M model fits entirely in GPU (100% offload, 3.1 GB). MedGemma at 7.4 GB exceeds the 6 GB VRAM limit, so Ollama splits it between GPU (60%) and CPU (40%). This CPU fallback is what makes it slower.

4. **Ollama automatically optimizes GPU offloading.** It loads as many layers as fit in VRAM and runs the rest on CPU. No manual configuration needed.
